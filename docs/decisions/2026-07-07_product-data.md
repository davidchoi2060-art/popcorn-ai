## ADR-001. 원본 보존은 product_imports 레이어로 분리

- **배경:** 06번(products 단일화)과 12번(원본 보존) 문서가 저장 기준에서 모순.
- **검토한 대안:** ① products에 원본 컬럼 유지 ② 원본 미보존 ③ 별도 imports 레이어(JSONB).
- **결정:** ③. 원본 행 전체를 product_imports.raw_row(JSONB)에 냉동.
- **버린 이유:** ①은 products 오염·모순 지속, ②는 룰 개선 시 재정규화 불가.
- **파급 효과:** products에서 category1~4/spec_raw 컬럼 제거. 재정규화는 imports에서 재실행.

## ADR-002. product_category_normalized 폐기, products 단일화

- **배경:** 분류 결과가 별도 테이블이면 모든 조회에 조인이 강제되고 정의처가 이원화됨.
- **검토한 대안:** ① 중간 테이블 유지 ② products 직접 보유.
- **결정:** ②. part_type/category_group/ai_candidate_yn을 products에, confidence/verified는 specs로 이동.
- **버린 이유:** ①은 두 문서 세대가 공존한 혼란의 원인이었고 조인 비용만 추가.
- **파급 효과:** 12번 문서 §9·§18 대체. 추천 쿼리 단순화.

## ADR-003. 덮어쓰기 우선순위는 locked_fields(JSONB)로 해결

- **배경:** 데이터 입수 경로 4개(CSV/소싱/수동/AI)의 충돌 시 승자 규칙 부재. 운영자 보정값이 CSV 재업로드에 증발하는 사고 위험.
- **검토한 대안:** ① margin_locked 단일 플래그 ② 필드별 잠금 매핑 테이블 ③ products.locked_fields JSONB 배열.
- **결정:** ③. 우선순위: 운영자 수동(locked) > 소싱 확정 > CSV > AI > 룰.
- **버린 이유:** ①은 입도가 거칠어 의미 불명, ②는 26,480개·운영자 2~3명 규모에 과설계.
- **파급 효과:** CSV/AI 파이프라인에 잠금 스킵 로직. ADM-CSV-010에 "보호됨 N건" 표기.

## ADR-004. specs 필드 잠금은 "specs." 네임스페이스로 products 한 곳 관리

- **배경:** 잠금 대상에 product_specs 소속 필드 포함. 관리 위치 이원화 시 한쪽만 조회하는 버그 위험.
- **검토한 대안:** ① specs에도 locked_fields 별도 보유 ② products 한 곳 + 접두 표기.
- **결정:** ②. 예: ["specs.length_mm", "sale_price"].
- **버린 이유:** ①은 잠금 조회 주체(상품 단위)와 불일치, 이중 관리 부채.
- **파급 효과:** 잠금 조회는 products.locked_fields 단일 지점.

## ADR-005. 정규화·룰 추출은 CSV 업서트 트랜잭션 내 동기 실행

- **배경:** 파이프라인 12단계는 정의됐으나 실행 시점·주체 미정.
- **검토한 대안:** ① 야간 배치 ② 수동 버튼 ③ 업서트 동기 실행.
- **결정:** ③. LLM 보강(2~3단계)만 업서트 후 비동기 잡.
- **버린 이유:** ①②는 "업로드했는데 추천에 안 보이는" 공백을 만들어 운영자가 버그로 인식.
- **파급 효과:** 업로드 완료 = 룰 추출·후보 반영 완료. 룰 기반·무LLM이라 성능 부담 없음.

## ADR-006. 부품 타입별 필수 연산 필드 매트릭스 신설

- **배경:** review_required_yn 판정 기준이 어디에도 없어 구현 불가 상태.
- **검토한 대안:** ① 전 필드 필수 ② 편집 패널 암시에 의존 ③ 타입별 필수/권장 매트릭스 명문화.
- **결정:** ③. ERD Ver 3.0 §5. 필수 NULL 시 review_required=true + 풀 제외.
- **버린 이유:** ①은 커버리지 전멸, ②는 판정 코드를 짤 수 없음.
- **파급 효과:** 호환성 검증 룰과 1:1 동기 개정 의무.

## ADR-007. 판매가 재계산은 자동 아닌 제안 + 승인

- **배경:** 매입가 변경(소싱/CSV) 시 판매가 처리 규칙 부재.
- **검토한 대안:** ① 마진 정책 자동 집행 ② 제안 배지 + 운영자 승인 ③ 미연동.
- **결정:** ②. 권장가 산출 → 배지 노출 → 승인 시 반영 + history(reason='margin_policy').
- **버린 이유:** ①은 마진 개선 기회를 시스템이 소진(가격 결정권 침해), ③은 정책 화면 무의미화.
- **파급 효과:** ADM-DSH-010 "가격 검토 대기 N건" 위젯. 마진 정책 = 제안 엔진으로 재정의.

## ADR-008. 스펙 정형 필드는 AI 보강 파이프라인으로 관리

- **배경:** 회사가 스펙·부품 상세까지 관리하는 것은 현실적으로 불가. 단, 스펙은 호환성 엔진 입력이라 환각 유입 시 "조립 불가 미출고" 약속 붕괴.
- **검토한 대안:** ① 전수 수동 입력 ② LLM 단독 채움 ③ 룰→원문LLM→지식LLM→교차검증 4단계.
- **결정:** ③. 원문 추출과 지식 조회의 독립 일치 시 verified 자동 승격, 불일치는 검수 큐.
- **버린 이유:** ①은 운영 불가, ②는 단일 경로 환각을 걸러낼 장치 부재.
- **파급 효과:** LLM 역할 원칙에 "③ 오프라인 데이터 보강(검증 게이트 필수)" 추가. ADM-PRD-020 검수 큐 화면 신규.

## ADR-009. 호환성 치명 필드는 verified 전 추천 풀 제외 + 재고 수량 미보유

- **배경:** length_mm 등은 동일 칩셋도 모델별 편차가 커 AI가 가장 틀리기 쉬운 필드가 곧 가장 치명적 필드. 재고 수량은 쇼핑몰이 원장.
- **검토한 대안:** (치명 필드) ① confidence 임계만 적용 ② verified 게이트 / (재고) ① 수량 동기화 ② 상태만 동기화.
- **결정:** 치명 필드(length_mm·gpu_max_mm·rated_watt·socket)는 verified 게이트, 재고는 상태만.
- **버린 이유:** confidence 단독은 독립 검증이 아니며, 수량 보유는 이중 원장 지옥(회원 DB와 동일 판단).
- **파급 효과:** 검수 큐 정렬은 노출 빈도순, "같은 모델 스펙 복제" 버튼으로 검수 경제성 확보.
