# S4. 주문 확인 · 결제 인계 — UI 설계 상세 프롬프트

**화면 ID(신규):** FR-REC-040 · **Screen 키:** `rec-cart` · **컴포넌트:** `RecCart`
**정체성:** 신뢰 판매자 · **역할:** 확정 구성을 기존 쇼핑몰 결제 레일로 넘기는 마지막 인계 지점
**디자인 시스템:** Vibrant Horizon (S0의 [A] 공통 규칙 승계)
**선행 결정:** ① 로그인 강제는 이 화면에서만(그 전까지 게스트 진행) ② 호환성 위험 시 진입 차단 ③ 결제 직전 최종 실재고 재확인

이 화면은 얇지만 신뢰의 마지막 관문이다. 결제 버튼을 누르기 직전에 **실재고를 한 번 더 확인**하고, "이 구성은 조립 가능함이 검증됐다"는 안심을 마지막으로 재강조한 뒤, 결제는 기존 팝콘PC 쇼핑몰 레일로 릴레이한다. 새 결제 시스템을 만들지 않는다.

> 사용 시 S0 파일의 **[A] 공통 규칙 블록을 이 프롬프트 앞에 그대로 붙여넣는다.** 아래는 S4 화면 상세만 담는다.

## [B] S4 화면 상세 — 주문 확인 · 결제 인계

```
위 [A] 공통 규칙을 적용해 "주문 확인 · 결제 인계" 화면을 만든다.
data-screen-id="FR-REC-040" data-domain="recommend".

전체 골격(단일 컬럼, 결제 집중형 · 산만한 요소 최소):
  ┌ 페이지 헤더(page-header): H1 "주문 확인"
  ├ 최종 신뢰 배지(cart__assurance)   ← 마지막 순간 안심
  ├ 장바구니 요약(cart__summary)      ← 구성 + 총액
  ├ 최종 실재고 재확인(cart__stock)   ← 결제 직전 재검
  ├ 로그인 게이트(cart__auth-gate, 게스트일 때만)
  └ 하단 sticky CTA 바(cart__cta): [부품 변경][결제하기]

────────────────────────────────────────
0) 구성 수정 진입 (얇게)
────────────────────────────────────────
- 결제 화면이므로 S1~S3의 상시 대화 바는 접어, 얇은 링크로 대체:
   "구성을 더 바꾸고 싶으세요? " + [부품 변경하기] data-action="goto-detail" → S3.
- 자유 대화 재추천은 이 단계에서 노출하지 않는다(결제 집중 · 이탈 방지).

────────────────────────────────────────
1) 최종 신뢰 배지 (cart__assurance) — 마지막 순간 안심
────────────────────────────────────────
- 강조 카드(primary 톤): H3 "이 구성은 조립 가능함이 검증됐습니다".
   호환성 5종 통과 미니 요약 data-repeat="compat_matrix"(✓ 텍스트 병기).
- 신뢰 요소 3칩(assurance__points, tertiary pill):
   [전문가 조립 · 검수] [당일 출고] [A/S 보증].
- 캡션: "결제 후 조립 검수를 거쳐 출고됩니다.".

────────────────────────────────────────
2) 장바구니 요약 (cart__summary)
────────────────────────────────────────
- 품목 리스트 data-repeat="cart_items"(부품 7종 + 조립 서비스 ASSEMBLY_SERVICE 포함):
   각 행 cart-item: 품목명 data-bind-item + 가격 + (부품별 수량 1 고정).
   * 조립 서비스는 별도 행으로 명시(숨기지 않음 — 신뢰 판매자 원칙: 비용 투명).
- 합계 영역(cart-summary__total):
   부품 합계 / 조립비 / 배송 / 총 결제금액
   <span class="price" data-bind="total_price">1,285,000원</span>(24px+ Bold primary, tabular).
   + 가격 기준 시각 <span data-bind="price_asof">3시간 전 기준</span>(S2·S3과 일관).
- (선택) 쿠폰·적립 입력 자리(cart__coupon, data-action="apply-coupon") — 기존 쇼핑몰 정책 따름.

────────────────────────────────────────
3) 최종 실재고 재확인 (cart__stock) — 결제 직전 재검
────────────────────────────────────────
- 결제 버튼 활성 전, 전 품목 재고를 한 번 더 대조한 상태 표시:
   정상(cart-stock--ok): "● 전 품목 재고 확보 — 바로 출고 가능"(primary, 텍스트 병기).
   품절 발생(cart-stock--soldout, hidden 기본, error):
     "● 방금 [부품]이 품절됐습니다. 대체 후 계속하시겠어요?"
     + [안전 대안으로 교체] data-action="fix-soldout" → S3 흐름 연결.
- 재확인 로딩(cart-stock--checking, hidden): "재고를 확인하는 중입니다...".

────────────────────────────────────────
4) 로그인 게이트 (cart__auth-gate) — 게스트일 때만 노출
────────────────────────────────────────
- 게스트 상태면 결제 전 인증 유도(그 전까지 게스트 진행 허용, 여기서 최초 강제):
   H3 "결제를 위해 로그인이 필요합니다".
   [로그인 / 회원가입으로 계속] data-action="open-auth"(기존 쇼핑몰 SSO 연동).
   캡션 "기존 팝콘PC 쇼핑몰 계정으로 로그인됩니다.".
- 로그인 완료 시 이 게이트는 숨기고 [결제하기] 활성.
- SSO 실패(auth__fail, hidden): "로그인에 실패했습니다. 다시 시도해 주세요."
   (게스트로는 결제 진행 불가 — 안내만).

────────────────────────────────────────
5) 하단 sticky CTA 바 (cart__cta)
────────────────────────────────────────
- [부품 변경] data-action="goto-detail" → S3.
- [결제하기] data-action="checkout" → 기존 쇼핑몰 결제 레일로 릴레이
   (확정 구성 페이로드 전달).
   ★ 비활성(btn--disabled) 조건: 호환성 위험 진입 / 미로그인 / 품절 미해결.
   각 조건별 사유 캡션 노출.

────────────────────────────────────────
예외 · 상태 UI (hidden 기본, Codex가 토글)
────────────────────────────────────────
- 위험 진입 차단(cart__blocked, hidden): S3에서 위험 상태로 넘어온 경우
   "호환성 위험이 남아 있어 결제로 진행할 수 없어요." + [부품 변경으로] → S3.
- 품절 발생(cart-stock--soldout): 위 3) 참조.
- 미로그인 결제 시도(auth__required, hidden): 로그인 게이트로 포커스 이동.
- 세션 만료(session__expired, hidden): "세션이 만료됐습니다. 구성은 보관돼 있어요."
   + 재인증 유도(입력·구성 보존).
- 결제 레일 오류(checkout__fail, hidden): "결제 연결에 문제가 있어요. 잠시 후 다시 시도해 주세요."
   구성 보존 + [다시 시도].

────────────────────────────────────────
반응형
────────────────────────────────────────
- 768px 이하: 단일 컬럼 유지(원래 집중형). 하단 CTA 바 고정(fixed),
   총 결제금액을 CTA 바에 함께 표기. 로그인 게이트는 전체폭 카드.

────────────────────────────────────────
접근성
────────────────────────────────────────
- 재고/호환/품절 상태 색 의존 금지(●·✓·⚠ 아이콘 + 텍스트 병기).
- [결제하기] disabled 시 사유를 aria-describedby로 연결.
- 인증 모달 focus-trap · aria-modal · ESC 닫기.
- 총 결제금액 aria-label로 단위 명시.

────────────────────────────────────────
산출 후 체크
────────────────────────────────────────
□ data-screen-id="FR-REC-040" data-domain="recommend"
□ 최종 신뢰 배지(호환 5종) + 조립·출고·A/S 신뢰 요소
□ 조립 서비스가 장바구니에 별도 행으로 명시(비용 투명)
□ 결제 직전 최종 실재고 재확인 상태(정상/품절) 존재
□ 게스트면 로그인 게이트, 로그인 후 [결제하기] 활성
□ [결제하기]가 위험/미로그인/품절 시 disabled, 사유 캡션
□ [결제하기]→외부 쇼핑몰 릴레이, [부품 변경]→S3 연결
□ 위험차단·품절·세션만료·결제오류 예외 hidden 포함
□ Inter+Pretendard, :root 토큰만, 한국어 통일
```

## 적용 노트 (기획자·개발자용)

1. **결제 직전 재고 재확인은 필수** — 신뢰 판매자의 마지막 방어선이다. S1~S3에서 통과했어도 결제 찰나에 품절될 수 있다(관리자 1초 차단, 동시 주문). checkout 직전 서버 재대조 없이 그냥 넘기면 "결제했는데 없는 부품"이 발생해 신뢰가 한 번에 무너진다. `cart-stock--soldout` 흐름을 반드시 살린다.
2. **결제 릴레이 페이로드 규약** — checkout 시 외부 쇼핑몰로 넘기는 확정 구성 페이로드(부품 코드·수량·조립 서비스·총액·recommendation_id)의 형식을 쇼핑몰 측과 확정해야 한다. 여기가 두 시스템의 경계이므로 계약을 명문화(07_api-spec에 결제 릴레이 항목 추가).
3. **인증은 SSO 재사용** — 새 회원 시스템을 만들지 말고 기존 쇼핑몰 회원 DB에 SSO 연동(auth-modal / FR-LND-020 재사용). 게스트로 여기까지 온 사용자의 세션·구성을 로그인 후에도 보존해야 이탈이 없다.
4. **조립비·배송비 투명 표기** — 신뢰 판매자 정체성상 조립 서비스와 배송비를 총액에 숨겨 넣지 말고 별도 행으로 드러낸다. "왜 이 가격인가"를 파는 정체성과 일관.
5. **여정 완결** — 이로써 사용자 여정 S0~S4가 모두 정의됐다. 남은 것은 (a) 전체 여정을 FigJam 플로우 다이어그램으로 박아 눈으로 검증, (b) 관리자 허브·고객문의 게시판(별도 트랙), (c) S0~S4에 흩어진 미정 결정 일괄 확정(success 토큰 / 비교 방향 / 커뮤니티 근거 소스 / 후보 풀 시각화 / 게임 무게 라인업).
