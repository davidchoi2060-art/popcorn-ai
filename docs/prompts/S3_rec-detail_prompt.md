# S3. 부품 변경 모드 — UI 설계 상세 프롬프트

**화면 ID(신규):** FR-REC-030 · **Screen 키:** `rec-detail` · **컴포넌트:** `RecDetail`
**대체 대상:** 기존 beg-detail(FR-BEG-110) + exp-detail(FR-EXP-110). 초급·고급 공용 단일 화면으로 통합
**정체성:** 신뢰 판매자 · **핵심 약속:** "바꿔도 조립 안 되는 구성은 안 나옵니다"
**디자인 시스템:** Vibrant Horizon (S0의 [A] 공통 규칙 승계)
**선행 결정:** ① 호환 100% 통과 대안만 노출 ② 위험 시 결제 잠금 ③ 연쇄 스왑은 정직하게 설명

이 화면은 사용자가 부품을 바꿔도 **조립 안전을 보증**한다. 호환 100% 통과한 대안만 보여주고, 스왑으로 호환성이 깨지면 원인과 연쇄 해결책을 정직하게 제시한 뒤 승인 시 자동 동시 교체한다. 기존 안전 대안·연쇄 스왑 구조를 그대로 재사용한다.

> 사용 시 S0 파일의 **[A] 공통 규칙 블록을 이 프롬프트 앞에 그대로 붙여넣는다.** 아래는 S3 화면 상세만 담는다.

## [B] S3 화면 상세 — 부품 변경 모드

```
위 [A] 공통 규칙을 적용해 "부품 상세 및 교체" 화면을 만든다.
data-screen-id="FR-REC-030" data-domain="recommend".

전체 골격:
  ┌ 상단 상시 대화 바(detail__composer-bar): 세션 내내 상주(자연어 교체 요청)
  ├ 페이지 헤더(page-header): H1 "부품 상세 및 교체"
  ├ 좌: 현재 구성 리스트(detail__parts)  ├ 우: 호환성 재검증 요약(detail__compat)
  ├ 안전 대안 선택(detail__alternatives, 부품 클릭 시)
  ├ 연쇄 스왑 안내(detail__chain, 위험 시)
  └ 하단 sticky CTA 바(detail__cta): [제안서로][장바구니]

────────────────────────────────────────
0) 상단 상시 대화 바 (detail__composer-bar)
────────────────────────────────────────
- S1·S2와 동일한 상시 입력 바 상주.
   <textarea placeholder="바꾸고 싶은 부품을 말로도 요청하세요. 예) 그래픽카드 한 단계 높여줘">
   + [보내기] data-action="refine-request".
- 입력 시 해당 부품의 안전 대안 탐색/적용 로직 실행(수기 클릭과 동일 결과).

────────────────────────────────────────
1) 현재 구성 리스트 (detail__parts)
────────────────────────────────────────
- 부품 카드 목록 data-repeat="components"(CPU/GPU/RAM/SSD/메인보드/파워/케이스):
   각 카드(part-card, surface-card 라운드 lg):
     · 부품명 data-bind-item + 가격.
     · 안전 대안 수 배지(part-card__alt-count, tertiary pill):
        "안전 대안 " + <span data-bind="alt_count">6</span> + "개".
     · [변경] <button data-action="swap-part" data-part="gpu">.
- 상단 총액 <span class="price" data-bind="total_price">1,250,000원</span>
   + 가격 기준 시각 <span data-bind="price_asof">3시간 전 기준</span> (S2와 일관).

────────────────────────────────────────
2) 호환성 재검증 요약 (detail__compat) — 신뢰 장치
────────────────────────────────────────
- 스왑할 때마다 5종 호환성을 재검증해 실시간 반영(S2 배지와 동일 구성):
   compat-matrix data-repeat="compat_matrix":
     CPU 소켓 / 메모리 규격 / 파워 정격 마진 / 쿨러 TDP / 케이스 실장.
   각 행 상태(텍스트 병기): 정상 ✓(primary) 또는 위험(error).
- 종합 상태 인디케이터(compat-status):
   · 정상: "● 조립 가능 — 호환 5종 통과"(primary, --ok).
   · 위험(compat-status--error, hidden 기본):
     "● 위험 — 파워 용량 부족"(error, 텍스트 병기).

────────────────────────────────────────
3) 안전 대안 선택 (detail__alternatives) — [변경] 클릭 시 노출
────────────────────────────────────────
- 인라인 패널 또는 모달(hidden 기본, swap-part로 노출):
   H3 "호환 가능한 대안" + 캡션 "현 구성과 호환 100% 통과한 제품만 보여드립니다.".
   대안 목록 data-repeat="alternatives":
     alt-item: 대안명 + 가격차(tertiary, "+80,000원"/"-50,000원") + 성능 변화 한 줄
       + [적용] data-action="apply-alternative" data-code="376681".
- 대안 없음(alternatives__empty, hidden):
   "지금 구성과 호환되는 안전한 대안이 없어요." + 사유 표기.

────────────────────────────────────────
4) 연쇄 스왑 안내 (detail__chain, hidden 기본) — 정직 설명
────────────────────────────────────────
- 위험을 유발하는 스왑(예: 고사양 GPU) 적용 시 노출:
   ⚠ H3 "연쇄 변경이 필요합니다".
   원인 본문 data-bind="chain_reason":
     "이 그래픽카드를 쓰려면 파워를 1000W 이상 등급으로 함께 바꿔야 조립이 가능합니다.".
   연쇄 대안 라디오 data-repeat="chain_options"
     ([시소닉 1000W +80,000원] 등, 가격차 병기).
   [연쇄 변경하고 계속하기] data-action="apply-chain-swap".
   캡션 "승인하시면 파워가 자동 교체되고 위험 상태가 해제됩니다.".
- 승인 시: 파워 자동 동시 교체 → compat-status--error 자동 해제 → 총액 갱신.

────────────────────────────────────────
5) 하단 sticky CTA 바 (detail__cta)
────────────────────────────────────────
- [제안서로 돌아가기] data-action="goto-result" → S2(변경분 반영된 제안서).
- [장바구니 담기] data-action="add-cart" → S4.
   ★ 위험 상태(compat-status--error)일 때 btn--disabled 락.
     캡션 "호환성 위험이 해결되어야 담을 수 있어요.".

────────────────────────────────────────
예외 · 상태 UI (hidden 기본, Codex가 토글)
────────────────────────────────────────
- 위험 결제 락(cta--locked, hidden): 장바구니 disabled + 사유 캡션.
- 대안 없음(alternatives__empty, hidden): 위 3) 참조.
- 스왑 후 품절 발생(swap__soldout, hidden):
   "방금 그 대안이 품절됐어요. 다른 대안을 보여드릴게요." (관리자 1초 차단과 연동).
- 로딩(loading-layer, hidden): "호환성을 다시 계산 중입니다...".

────────────────────────────────────────
반응형
────────────────────────────────────────
- 768px 이하: 좌(구성)/우(호환성) 2열→세로 스택. 호환성 요약을 상단 sticky로.
   안전 대안은 전체폭 모달(바텀시트)로. 하단 CTA 바 고정(fixed).

────────────────────────────────────────
접근성
────────────────────────────────────────
- 호환성 상태 색 의존 금지(정상 ✓ / 위험 ⚠ 아이콘+텍스트 병기).
- 대안 모달은 focus-trap, ESC 닫기, aria-modal.
- 위험 발생 시 aria-live로 "호환성 위험 발생, 결제가 잠겼습니다" 정중 안내.
- [변경]·[적용]·[연쇄 변경] 모두 <button> + 명확한 라벨.

────────────────────────────────────────
산출 후 체크
────────────────────────────────────────
□ data-screen-id="FR-REC-030" data-domain="recommend"
□ 부품별 [변경](swap-part) + 안전 대안 목록(호환 100% 통과만)
□ 스왑 시 호환성 5종 재검증이 반영되는가
□ 연쇄 스왑 안내(apply-chain-swap)와 승인 시 자동 해제 흐름
□ 위험 상태에서 [장바구니] disabled 락
□ 대안없음·품절·위험락·로딩 예외 hidden 포함
□ [제안서로]→S2, [장바구니]→S4 연결
□ Inter+Pretendard, :root 토큰만, 한국어 통일
```

## 적용 노트 (기획자·개발자용)

1. **안전 대안·연쇄 스왑 백엔드 재사용** — 기존 `POST /api/swap/candidates`(호환 100% 통과 대안만)와 `POST /api/swap/validate`(연쇄 검증)를 그대로 쓴다. UI만 새 토큰으로 갈아입히는 것이므로 로직 리스크는 낮다.
2. **스왑 후 근거·제약 동기화** — 부품을 바꾸면 S2 근거 리포트와 총액·성능이 달라진다. [제안서로 돌아가기] 시 변경분이 반영된 근거로 갱신되게, 스왑 결과를 세션 상태(제약·구성)에 되먹여야 한다. 스왑이 S2와 따로 놀면 "제안서엔 옛 부품이 남아있는" 불일치가 난다.
3. **품절 실시간 반영** — 관리자의 품절/단종 1초 차단(EX-ADM-01)이 안전 대안 목록에도 적용돼야 한다. 사용자가 대안을 고르는 찰나 품절될 수 있으니 `swap__soldout` 처리를 반드시 살려둔다.
4. **success 토큰 부재(S2와 동일 이슈)** — 정상/통과 표기를 primary로 갈지 `--color-success`를 추가할지 S2·S3에 걸쳐 한 번에 결정. 화면마다 다르면 안 된다.
5. **후보 풀 카운터의 S3 변형** — S1의 전체 후보 풀 카운터 대신, S3에서는 부품별 "안전 대안 N개" 배지로 축소 표현했다(alt_count). 전체 풀을 다시 보여주면 맥락이 어긋나므로 이 축소가 맞다. 이 방향으로 갈지만 확인.
