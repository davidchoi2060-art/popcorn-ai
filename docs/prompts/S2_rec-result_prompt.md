# S2. 견적 제안서 — UI 설계 상세 프롬프트

**화면 ID(신규):** FR-REC-020 · **Screen 키:** `rec-result` · **컴포넌트:** `RecResult`
**대체 대상:** 기존 beg-result(FR-BEG-090) + exp-result(FR-EXP-090). 근거 리포트·비교견적을 별 화면으로 쪼개지 않고 이 한 장에 흡수
**정체성:** 신뢰 판매자 · **히어로 약속의 증명 화면** — "모든 견적에는 이유가 있습니다"
**디자인 시스템:** Vibrant Horizon (S0의 [A] 공통 규칙 승계)
**선행 결정:** ① 결과·근거·비교를 한 장짜리 스크롤 제안서로 통합 ② 호환성 배지를 가격보다 먼저 ③ 비교는 가격 레이스가 아니라 정직 프레임

이 화면은 신뢰 판매자가 내는 **한 장짜리 제안서**다. 슬라이드처럼 쪼개지 말고, 앵커 탭으로 이동하는 하나의 스크롤 문서로 만든다. 결과(구성)·근거·비교가 위→아래로 흐른다.

> 사용 시 S0 파일의 **[A] 공통 규칙 블록을 이 프롬프트 앞에 그대로 붙여넣는다.** 아래는 S2 화면 상세만 담는다.

## 적용 노트 (기획자·개발자용)

1. **근거 리포트는 엔진 산출값이어야 한다** — reason_items를 LLM에 "이 구성 설명해줘"로 사후 생성시키면 신뢰 판매자 정체성이 무너진다(창작·환각 위험). 결정론 엔진이 각 부품을 **어떤 하드필터·점수·가중치로 골랐는지**를 구조화해 그대로 바인딩하고, LLM은 그 사실을 사람 말투로 다듬는 데만 쓴다. 즉 근거의 출처는 엔진, 문장 다듬기만 LLM.
2. **success 토큰 부재** — Vibrant Horizon엔 초록 계열이 없다. '통과 ✓'를 초록으로 임의 생성하지 말고 primary(#005ab3)로 통일하거나, `--color-success`를 팔레트에 정식 추가할지 결정해야 한다. 위험은 error(#b31b25) 유지. 어느 쪽이든 색+텍스트 병기 원칙은 지킨다.
3. **비교견적 데이터 출처를 먼저 정하라** — 최저가 비교(compare_rows)의 '최저가'가 어디서 오는가. 다나와 직접 크롤링은 ToS 리스크가 있으니 제휴 API가 최선이고, 불가하면 "시장 최저가(N시간 전 기준)"로 시점을 정직 표기하거나, 이 섹션을 '외부 견적 진단(4-2)' 중심으로 축소하는 것도 방법이다. 정체성상 4-2(진단)가 4-1(가격 비교)보다 우리에게 유리하다.
4. **가격 신선도(#3)** — price_asof·price--stale은 연출이 아니라 실제 갱신 파이프라인과 연결돼야 한다. "실시간"이라 써놓고 하루 지난 값이면 신뢰가 더 깨진다. 갱신 주기를 정하고 그대로 표기.
5. **아직 안 정한 것** — (a) 비교견적을 가격표 중심으로 갈지 진단 중심으로 갈지(위 3번), (b) 근거 태그의 '커뮤니티 최다 추천' 근거를 어디서 확보할지(실제 커뮤니티 데이터 소스 vs 내부 판매·스왑 통계 기반). 이건 데이터 확보 가능성과 직결되니 기획·데이터와 합의 필요.
