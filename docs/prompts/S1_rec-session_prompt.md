# S1. 대화형 견적 세션 — UI 설계 상세 프롬프트 (개편의 심장)

**화면 ID(신규):** FR-REC-010 · **Screen 키:** `rec-session` · **컴포넌트:** `RecSession`
**대체 대상:** 기존 beg-step1~4(FR-BEG-010~050) + exp-step1~5(FR-EXP-010~060) 9개 스텝 화면을 이 1개로 통합
**정체성:** 신뢰 판매자 · **디자인 시스템:** Vibrant Horizon (S0의 [A] 공통 규칙 승계)
**선행 결정:** ① 대화가 주 / 칩은 가속기 ② 초급·고급 하드분기 삭제 ③ 기어(추천질문 ⇄ 직접입력) 사용자 강제 전환

이 화면은 초급/고급을 화면으로 가르지 않고, **대화 하나에서 두 기어로 흡수**한다. 매 답변마다 실재고 후보 풀이 눈앞에서 줄어드는 것을 보여주며 신뢰를 적립하고, 최종적으로 결정론 엔진이 3구성을 만들 조건이 차면 S2로 넘긴다.

> 사용 시 S0 파일의 **[A] 공통 규칙 블록(Vibrant Horizon 토큰 + 마크업 계약)을 이 프롬프트 앞에 그대로 붙여넣는다.** 아래는 S1 화면 상세만 담는다.

## [B] S1 화면 상세 — 대화형 견적 세션

```
위 [A] 공통 규칙을 적용해 "대화형 견적 세션" 화면을 만든다.
data-screen-id="FR-REC-010" data-domain="recommend".

전체 3분할 골격(데스크톱):
  ┌ 상단 바(session__topbar): 기어 토글 + 진행 요약
  ├ 좌·중앙(session__stream ≈62%): 대화 스트림
  ├ 우측(session__aside ≈38%, sticky): 후보 풀 카운터 + 현재 제약 요약
  └ 하단(session__composer): 상시 자유 입력 바 + [견적 확정] CTA

────────────────────────────────────────
1) 상단 바 (session__topbar) — 기어 전환의 집
────────────────────────────────────────
- 좌측 브랜드 축약 + "AI 견적 세션".
- ★ 기어 토글(gear-toggle, 세그먼티드 컨트롤, role="radiogroup"):
    [추천 질문 받기] data-action="switch-gear" data-gear="ask" (기본 활성 --active)
    [직접 다 입력]   data-action="switch-gear" data-gear="manual"
  캡션(label, on-surface-variant): "언제든 바꿀 수 있어요. 지금까지 고른 조건은 유지됩니다."
- 우측: 진행 마이크로 요약 "조건 3개 반영됨"(data-bind="constraint_count").
- ※ 기어 전환은 리셋이 아니다. 아래 4)의 제약 요약이 단일 원천이며,
   두 기어 모두 같은 제약 상태를 읽고 쓴다. (구현 함정 — 반드시 준수)

────────────────────────────────────────
2) 대화 스트림 (session__stream) — 주 경로
────────────────────────────────────────
- aria-live="polite" 영역. 새 AI 메시지가 아래로 쌓이고 자동 스크롤.
- 메시지 버블 data-repeat="messages":
   · AI(msg msg--ai): surface-container 틴트 배경, 좌측 정렬, 아바타 소형.
   · 사용자(msg msg--user): primary 옅은 틴트, 우측 정렬.
- ★ 한 번에 질문 하나(AI). 질문 버블 아래에 빠른 답변 칩(answer-chips):
   <button class="chip" data-action="quick-answer" data-q="game_weight"
     data-value="popular">인기게임</button> 형태, pill.
   * 칩을 눌러도 되고, 하단 입력 바에 자유롭게 쳐도 된다(칩은 가속기).
   * 칩 위계는 입력 바보다 낮게 — 자유 입력이 주 경로임을 시각적으로 종속.

- 질문 시퀀스(ask 기어 기본 흐름, 더미):
   Q1 용도 — 칩 [게임][사무][영상편집][방송]
   Q2 (게임 선택 시) 게임 무게 — 칩 [가벼운(롤·오버워치)][인기(배그·로스트아크)][무거운(사이버펑크·AAA)]
        + 구체 게임 자유 입력 유도
   Q3 예산 — 칩 [80만대][100만대][150만대] + "직접 입력" 유도
   Q4 취향(다중) — 칩 [조용한][화이트][작은 크기][가성비 최우선][화려한 LED]
   Q5 확인 — "이대로 후보를 좁혔습니다. 견적 확정할까요?"
   * 각 질문은 사용자 답변에 따라 동적 분기(게임 아니면 Q2 생략 등).

- ★ 자유 점프·수정: 사용자가 흐름 도중 하단 입력 바에
   "그냥 조용한 걸로", "예산 20만 올릴게", "화이트로 바꿔줘" 등을 입력하면
   해당 제약을 즉시 갱신하고 AI가 반영 확인 버블로 응답.
   (이것이 상단 별도 요청창을 두지 않는 이유 — 세션 자체가 그 역할)

────────────────────────────────────────
3) 직접 입력 기어 (session__poweredit, 기본 hidden)
────────────────────────────────────────
- [직접 다 입력] 전환 시 스트림 상단에 큰 자유 입력 영역이 열린다:
   <textarea class="poweredit__input"
     placeholder="스펙을 한 번에 입력하세요. 예) AMD 9600X, RTX4070, DDR5 32G, 예산 200만, 3열 수랭, 조용하게">
   + [분석하기] data-action="parse-spec".
- 파싱 후: 전체 질문지를 다시 묻지 않고, **부족한 필수값만** 소형 칩으로 확인
   (예: 예산 미입력 시 예산 칩만 노출). data-repeat="missing_fields".
- ask 기어로 되돌아가도 파싱된 제약은 그대로 유지(4의 요약에 반영됨).

────────────────────────────────────────
4) 후보 풀 카운터 + 제약 요약 (session__aside) — 신뢰 장치
────────────────────────────────────────
- ★ 후보 풀 카운터(pool-counter, sticky 상단):
   H2 라벨 "조건 통과 실재고".
   대형 수치 <span class="pool-counter__num" data-bind="candidate_count">1,240</span> + "대".
   (Inter tabular, headline-lg급, primary. 값 변할 때 숫자 틱다운 + 축소 애니메이션)
   보조: "전체 " + <span data-bind="total_stock">2,480</span> + "대 중" .
   진행 바 or 집합 도트 축소 시각화(pool-counter__viz) — 색 의존 말고 수치 병기.
   * 각 답변/수정 = 하드필터(재고→호환성→예산) 즉시 반영해 재집계.
   * 성능: 매 입력마다 풀 쿼리 금지. 사전 집계/캐시 카운트 사용(적용 노트 참조).

- 현재 제약 요약(constraint-summary, data-repeat="constraints"):
   각 제약을 pill로 표시 + 개별 [수정]/[삭제]:
   <span class="constraint" data-constraint="budget">예산 100~130만
     <button data-action="edit-constraint" data-constraint="budget">수정</button></span>
   * 여기가 제약의 단일 원천. 두 기어·자유 입력이 모두 이 목록을 갱신.
   * [수정] → 입력 바에 해당 항목 편집 컨텍스트 주입.

────────────────────────────────────────
5) 하단 상시 입력 바 + 확정 CTA (session__composer)
────────────────────────────────────────
- 상시 자유 입력(항상 노출, 세션 내내):
   <textarea class="composer__input" placeholder="추가 요청이나 수정 사항을 입력하세요">
   + [보내기] data-action="send-message".
- ★ [견적 확정] data-action="confirm-estimate" (기본 btn--disabled):
   엔진이 3구성 생성 최소 조건 충족 시 활성(--primary).
   파워유저는 첫 입력만으로 즉시 활성될 수 있음.
   캡션: "지금 조건으로 가성비·추천·고성능 3구성을 만들 수 있어요."
- confirm → 로딩 레이어 노출 후 S2(견적 제안서)로 전환. 제약·raw_text 이관.

────────────────────────────────────────
예외 · 상태 UI (hidden 기본, Codex가 토글)
────────────────────────────────────────
- ★ 후보 0대(session__zero, hidden): 신뢰의 결정적 순간.
   "지금 조건에선 후보가 0대예요."
   + 원인 지목 data-bind="zero_reason"("예산 50만 + RTX4090 조합이 후보를 0으로 만들었어요")
   + 완화 제안 data-repeat="relaxations"
     ([GPU 한 단계 낮추기 → 12대] data-action="apply-relaxation" data-relax="gpu_down").
- 모순 요구(session__conflict, hidden): AI 재확인 버블
   "조용한 구성과 3열 수랭 최대 RPM은 상충할 수 있어요. 어느 쪽을 우선할까요?" + 선택 칩.
- 파싱 실패(session__parse-fail, hidden): "일부를 이해하지 못했어요. 이 부분만 다시 알려주세요."
   입력값 보존 + [다시 시도] data-action="retry".
- 로딩(loading-layer, hidden): "실재고에서 최적 구성을 조율 중입니다...".
- (기존 예외 승계) 3사 실패 LLM_ALL_FAILED / 한도초과 RATE_LIMIT_EXCEEDED /
   비용차단 COST_CIRCUIT_OPEN 안내 레이어도 hidden으로 포함.

────────────────────────────────────────
반응형
────────────────────────────────────────
- 768px 이하:
   · session__aside → 화면 상단 sticky 스트립으로 이동.
     후보 풀 카운터는 축약 수치, 제약 요약은 가로 스크롤 pill.
   · session__stream 폭 100%.
   · session__composer는 화면 하단 고정(fixed), [견적 확정]은 입력 바 우측 or 상단 sticky.
   · 기어 토글은 상단 바에서 아이콘+짧은 라벨로 축약.

────────────────────────────────────────
접근성
────────────────────────────────────────
- 대화 스트림 aria-live="polite", 새 AI 질문 낭독.
- 기어 토글 role="radiogroup", 각 옵션 aria-checked.
- 후보 풀 카운터 변화는 aria-live로 정중히 알림(과다 낭독 방지 debounce).
- 모든 칩·버튼 <button> + 텍스트 라벨. 색 의존 금지(제약·위험 텍스트 병기).

────────────────────────────────────────
산출 후 체크
────────────────────────────────────────
□ data-screen-id="FR-REC-010" data-domain="recommend"
□ 기어 토글(switch-gear) 존재 + 전환 시 제약 요약이 유지되는 구조인가(리셋 아님)
□ 후보 풀 카운터 candidate_count에 data-bind + 축소 애니메이션 자리
□ 제약 요약이 단일 원천이고 두 기어·자유입력이 모두 갱신하는가

□ [견적 확정]이 기본 disabled, 최소 조건 충족 시 활성
□ 후보 0대·모순·파싱실패·로딩 예외 UI가 hidden으로 포함
□ 초급/고급 분리 화면이 없고 단일 세션인가
□ Inter+Pretendard, :root 토큰만, 한국어 통일
```

## 적용 노트 (기획자·개발자용)

1. **사이트맵 갱신 필요** — 이 화면은 신규 `FR-REC-010` / `rec-session`으로, 기존 beg-step1~4·exp-step1~5를 대체한다. `01_sitemap.md`·`02_file-list.md`에 REC 도메인을 추가하고 구 스텝 화면을 "통합됨"으로 표기해야 문서-구현 불일치가 안 생긴다.
2. **후보 풀 카운터용 신규 엔드포인트** — 매 답변마다 `POST /api/recommend`를 때리면 비용·지연이 폭발한다. LLM 없이 DB만 도는 경량 카운트 API(예: `POST /api/candidates/count`, 입력=현재 제약, 출력=하드필터 통과 수)가 필요하다. 이건 결정론 엔진(피벗 1의 validator 승격분)만으로 처리되므로 비용 0. 사전 집계 테이블/캐시로 응답 100ms 이내를 목표.
3. **제약 요약 = 상태의 단일 원천** — ask 기어의 칩 선택, manual 기어의 파싱 결과, 자유 입력 수정이 모두 이 하나의 constraints 객체를 갱신한다. 기어 전환은 뷰만 바꾸고 상태는 공유. 이 계약을 어기면 "직접 입력으로 바꿨더니 방금 고른 게 날아갔다"는 최악의 이탈이 난다.
4. **확정 최소 조건 정의 필요** — [견적 확정]이 언제 활성화되는지(예: 용도+예산 2개면 충분한지, 취향까지 필요한지)를 엔진 쪽과 합의해 명문화해야 한다. 파워유저 즉시 활성 케이스도 이 규칙에 포함.
5. **아직 안 정한 것** — 후보 풀 축소 시각화의 형태(숫자 틱다운만 vs 집합 도트/막대 애니메이션), 그리고 게임 무게 3분류의 대표 게임 라인업 확정(마케팅·데이터와 합의).
