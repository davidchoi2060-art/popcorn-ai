## 프로젝트 한 줄
자체 부품 풀 기반 PC 견적·판매 시스템. 정체성 = **신뢰 판매자**("조립 불가 PC는 안 나온다 + 왜 이 부품인지 설명"). 히어로: "모든 견적에는 이유가 있습니다."

## 지금 단계 = 모델링 (절대 규칙)
- 산출물은 **클릭 가능한 UI 목업(HTML/CSS/JS)** 뿐.
- **금지:** 백엔드·API 서버·DB 스키마/마이그레이션·실제 LLM 연동·서버 배포·빌드 시스템/npm 도입.
- 이 규칙과 충돌하는 요청이 오면 **코드 짜지 말고 먼저 물어라**: "지금 모델링 단계인데 뒷단을 시작할까요?"
- 이유: 과거 UI 미확정 상태로 뒷단부터 지어 실패 반복. UI가 손으로 확정될 때까지 뒤는 건드리지 않는다.

## 작업 방식 (속도 핵심 — 이게 느림을 잡는다)
1. **탐색 최소화.** 작업 전 전체 코드베이스를 훑지 마라. 필요한 파일만 열어라. 어떤 파일을 볼지 모르면 먼저 물어라.
2. **한 번에 하나.** 큰 작업을 한 요청에 몰지 마라. "S2 전체" 대신 "S2 섹션1"처럼 쪼개라.
3. **계획 먼저, 코드 나중.** 파일을 만들기 전 3~5줄로 접근을 요약하고 승인받아라(Plan Mode 선호).
4. **끝나면 자체 검증.** 목업은 브라우저에서 열리는지·JS 콘솔 에러 없는지 스스로 점검 후 보고.
5. **말은 짧게.** 턴 사이 설명은 최소, 최종 보고는 요점만. 장황한 서술 금지.
6. **세션은 일회용.** 결정은 이 파일/`docs/decisions/decision-log.md`에 기록. 긴 세션에 기억을 의존하지 마라. 컨텍스트가 길어지면 `/clear` 후 재시작.

## 디자인 계약
- 토큰 단일원천 = `design-system/tokens.css` `:root`. 임의 HEX/px 금지.
- 서체: Inter(라틴·숫자) + Pretendard(한글). 숫자는 tabular.
- 색: primary `#005ab3`/hover `#0082fd`, tertiary `#853d97`, error `#b31b25`. **초록 없음** → 통과=primary, 위험=error, 항상 텍스트 병기.
- **아이콘: 단색 currentColor 인라인 SVG만.** 풀컬러 팩(fluent-color/streamline) 금지.
- 마크업: `data-screen-id`/`data-domain`/`data-bind`/`data-repeat`/`data-action`, 예외는 `hidden`. 한국어 UI, 768px 세로 스택.

## 상태 관리 원칙 (S1에서 확립)
- 제약은 **객체 하나가 단일 원천**. 칩·자유입력·파싱이 전부 그 하나만 갱신.
- 기어/뷰 전환은 상태를 리셋하지 않는다.
- 후보 풀 카운터 = 완성품 대수 아님. **전체 부품(26,480개)에서 조건 통과 부품 수.**
