# Claude Code 속도 운영 가이드 — 팝콘PC AI

> 이 파일은 사람이 읽는 운영 가이드다. **CLAUDE.md에 넣지 마라**(매 턴 로드되어 오히려 느려짐).
> 근거: Anthropic 공식 베스트프랙티스 + 커뮤니티 최적화 사례(2026).

## 왜 느린가 — 진짜 원인 (냉정하게)

"Claude가 느리다"의 원인은 대부분 프롬프트 문구가 아니라 **컨텍스트 관리**다.
Anthropic 공식: "가장 중요한 제약은 컨텍스트 창이 빨리 차고, 차면 성능이 떨어진다는 것."

느림·낭비의 3대 원인:
1. **매 작업마다 전체 코드베이스 탐색** — 안 봐도 될 파일을 계속 읽음.
2. **큰 작업을 한 번에** — 한 요청에 "S2 전체" 몰기 → 길고 자주 실패.
3. **긴 세션 누적** — 컨텍스트가 쌓여 점점 느려지고 헤맴.

이 셋을 잡으면 체감 속도·토큰이 크게 준다. CLAUDE.md 문구 튜닝은 그다음 문제다.

## 즉시 적용할 습관 5가지

1. **작게 쪼개서 시켜라.** "S2 견적 제안서 전체" ❌ → "S2 섹션1 구성요약만" ✅.
   한 번에 하나. 성공률↑, 재작업↓.
2. **탐색 범위를 지정하라.** "s1-session.html만 보고 고쳐" 처럼 파일을 못박아라.
   Claude가 알아서 폴더를 훑게 두면 토큰이 샌다.
3. **세션을 자주 비워라.** 작업 하나 끝나면 `/clear`. 컨텍스트가 길어지면 성능이 떨어진다.
   결정은 반드시 `decision-log.md`에 남겨 세션에 의존하지 마라(세션은 일회용).
4. **Plan Mode를 써라.** 큰 변경 전 계획을 먼저 세우게 하고(plan.md), 승인 후 실행.
   무계획 실행은 성공률 ~33%. 계획이 성공률을 끌어올린다.
5. **모델을 작업에 맞춰라.** 기계적 작업(이름변경·포맷)은 가볍게, 판단이 필요한 설계·디버깅은 강한 모델.
   가능하면 계획=강한 모델 / 구현=빠른 모델로 분리(opusplan류).

## effort(사고량) 조절

- 길게 생각할수록 결과는 좋지만 느리고 토큰을 쓴다.
- **간단·기계적 작업은 effort를 낮춰** 지연을 줄여라(`/effort` 또는 설정).
- 어려운 설계·상태관리 로직만 높은 effort.

## CLAUDE.md 자체를 가볍게 (이미 반영됨)

- CLAUDE.md는 **매 턴 전체가 로드**된다. 5,000토큰짜리면 한 마디 치기 전에 매번 5,000토큰을 문다.
- 그래서 이 프로젝트 CLAUDE.md는 **~500토큰**으로 짧게 유지했다.
- 상세 배경·절차는 CLAUDE.md가 아니라 `docs/`·`HANDOFF.md`에 두고, **필요할 때만 읽게** 한다(just-in-time).
- CLAUDE.md에 계속 규칙을 덧붙이고 싶은 유혹을 참아라. 길어지면 느려진다.

## 이 프로젝트에 맞는 프론트엔드 검증 루프

목업 작업은 "만들고 → 눈으로 확인 → 고침"의 짧은 루프가 빠르다.
- 만든 뒤 **브라우저로 직접 열어** 확인하라고 지시(빌드 없음).
- 스크린샷을 찍어 붙여주면(Windows: **Alt+V**) Claude가 시각 차이를 보고 수정한다.
- "JS 콘솔 에러 없는지 확인 후 보고"를 습관화.

## 세션 시작 템플릿 (복붙용)

```
HANDOFF.md, CLAUDE.md, docs/decisions/decision-log.md만 읽고
현재 상태를 5줄로 요약해줘. 전체 폴더는 스캔하지 마.
그 다음 오늘 할 작업 하나만 제안해줘.
```

## 하지 말 것

- 전체 폴더 스캔 요청("프로젝트 전체 파악해줘") — 토큰 폭발.
- 한 요청에 여러 화면 동시 작업.
- 긴 세션을 유일한 기록으로 삼기 — 반드시 파일로 덤프.
- CLAUDE.md에 배경·역사·상세 절차 욱여넣기.

## 측정하라 (추측 금지)

- 대표 작업 하나를 정해 토큰/시간을 기록 → 위 습관 적용 후 같은 작업 재측정 → 비교.
- 구독(Pro/Max)이면 claude.ai → Settings → Usage에서 사용량 확인.
- 측정 없이 "빨라진 것 같다"는 착각이다.
