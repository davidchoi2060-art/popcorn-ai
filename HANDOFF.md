## 0. 30초 요약 (가장 먼저 읽을 것)

- **무엇을 만드나:** 자체 실재고 부품 기반 PC 견적·판매 시스템(팝콘PC AI). 정체성은 **"신뢰 판매자"**.
- **지금 단계:** **모델링.** 산출물은 클릭 가능한 UI 목업뿐. 백엔드/DB/서버/AI연동은 **만들지 않는다.**
- **왜 이 규칙이 절대적인가:** 과거 이 프로젝트는 UI가 확정되지 않은 채 뒷단부터 구축해, 불명확하게 굳고 결국 울며 겨자 먹기로 오픈하는 실패를 반복했다. 그래서 **UI가 손으로 검증·확정되기 전엔 어떤 뒷단 작업도 시작하지 않는다.**
- **지금 진행률:** 게이트웨이(index) 초안 + S1(대화형 세션) 목업 "진짜 도는" 버전 완료. S0는 재정비 필요, S2~S4는 프롬프트만 있음.
- **다음 할 일:** S1을 손으로 최종 판정 → 통과 시 S0 재정비 → S2 목업. (아래 8절)

## 1. 프로젝트 정체성

**팝콘PC AI** — 기존 팝콘PC 쇼핑몰(서울 용산)에 붙는 부품 기반 AI 견적·판매 시스템.

**핵심 포지셔닝: "신뢰 판매자"** (가격 비교 포털이 아님)
- 파는 것은 부품이 아니라 안심: "조립 불가능한 PC는 우리에게선 안 나온다 + 왜 이 조합인지 다 설명한다."
- 히어로 헤드라인: **"모든 견적에는 이유가 있습니다."**

**핵심 아키텍처 사상 (반드시 유지):**
- 추천 = **결정론적 룰/스코어 엔진**이 자체 부품 풀 위에서 수행. 같은 입력 + 같은 재고 → 같은 결과(재현성).
- **AI(LLM)의 역할은 입력 파싱 + 근거 설명뿐.** 견적을 생성하지 않는다.
- 이유: 재현성 확보 + 근거가 창작이 아닌 사실이 되게 함(신뢰 판매자 정체성의 근거).
- (과거 3사 LLM 병렬로 견적을 "생성"하던 구조는 폐기. validator를 주엔진으로 승격하는 방향.)

**3개 MVP (독립 서비스로 병렬 포지셔닝, 엔진 ①을 2·3이 공유):**
1. 팝콘PC 단독 AI 견적 — **가장 먼저 오픈, 코어. 현재 모든 목업의 대상.**
2. AI 제안 + 용산 도매 최저가
3. 입점 부품 가격비교 포털 — **외부 크롤링 미사용, 입점사 가격만**(법적 리스크 제거 완료)

**환경 구분:** 지금 다루는 GCP 서버 = **개발 서버**. 운영 서버는 별도 시스템으로 추후 구현.

## 2. 확정된 UX 결정 (번복 금지, 근거 있음)

- 대화 입력이 **주**, 칩은 대화 안에서 빠르게 답하는 **가속기**.
- 입력창 **전면**, 세일즈 요소는 **하단**.
- 초급/고급 하드분기 **삭제** → 하나의 대화 세션에 **기어 2개**(추천질문 ↔ 직접입력), 사용자가 언제든 강제 전환.
- **기어를 전환해도 이미 파싱된 제약은 유지**(상태 보존). 이게 S1의 최대 구현 함정.
- 결과·근거·비교를 **한 장짜리 스크롤 제안서(S2)**로 통합.
- 호환성 배지를 **가격보다 먼저** 노출.
- 비교견적은 가격 레이스가 아니라 **정직 프레임**("왜 이 가격/왜 우리가 더 안전한가").
- 후보 풀 카운터 = **완성 PC 대수가 아니라 "전체 부품(26,480개)에서 조건 통과 부품 수"**. (개념 교정 완료)
- 게이트웨이(index) = **내부자 전용 바로가기 허브**(직원/운영자/개발자). 세일즈 아님, 속도·명료함 우선.

### S1 목업에 실제 구현된 3대 핵심 (검증됨)
1. **상태 단일 원천:** `session.constraints` 객체 하나. 칩·자유입력·파싱·완화가 전부 `setConstraint()`만 경유. 기어 전환 시 constraints 미변경(=조건 유지).
2. **실시간 후보 풀 카운터:** 답변마다 부품 수 틱다운(26,480→…) + 도트 40개 축소 + 진행바 + 델타 표시.
3. **단색 currentColor 인라인 SVG 아이콘:** 풀컬러 팩(fluent-color/streamline) 전량 제거. 토큰 색 상속.

## 4. 디자인 계약 (반드시 준수)

**디자인 시스템: Vibrant Horizon.** 토큰 단일 원천 = `design-system/tokens.css`의 `:root`.
- 모든 목업은 이 파일을 링크해 토큰만 사용. 임의 HEX/px, 인라인 스타일 지양.
- 서체: **Inter(라틴·숫자) + Pretendard(한글)** 페어링. Inter 단독 금지(한글 깨짐). 숫자·가격은 Inter tabular.
- primary `#005ab3`(액션) / `#0082fd`(hover) / tertiary `#853d97`(근거·태그) / error `#b31b25`(위험).
- **success(초록) 토큰 부재** → '통과/정상'은 primary로, '위험'은 error로, **항상 텍스트 병기**(색 의존 금지).
- 라운드: 칩 pill, 카드 1rem. 앰비언트 그림자.
- **아이콘: 단색 currentColor 세트만**(Lucide 권장 또는 인라인 SVG). 풀컬러 팩 금지(토큰 색 통제 불가·톤 불일치).

**마크업 계약 (Stitch 규약 승계):**
- 최상위 `data-screen-id`, `data-domain`. 동적값 `data-bind`, 반복 `data-repeat`, 버튼·링크 `data-action`, 예외/상태 UI는 `hidden`.
- 시맨틱 태그, `alt`, `label` 연결, 접근성(aria) 준수. 한국어 UI(기술 약어만 영문). 768px 이하 세로 스택.

## 6. 인프라 현황 (개발 서버, GCP 서울)

> 상세 절차·명령어는 `docs/infra/10_infra-deploy-gcp.md`, 민감 식별자는 `docs/infra/GCP_SETUP.md` 참조.

- **리전:** asia-northeast3(서울) 고정 (지연 + PIPA).
- **VM:** Compute Engine `popcorn-app`, e2-small, Ubuntu 24.04. Nginx/Node20/PM2/Redis/certbot 설치 완료.
- **DB(A안):** Cloud SQL `popcorn-db`, PostgreSQL 16, db-g1-small, 자동 백업+PITR, 삭제방지. **연결 검증 완료(`SELECT 1` 성공).**
- **연결 방식:** 개발 검증은 공개 IP + 승인된 네트워크(VM IP /32) + `sslmode=require`. (운영 전 Auth Proxy로 전환 권장 — Auth Proxy는 VM 액세스 스코프 이슈로 보류.)
- **미완:** DB 앱 유저·확장 생성(`popcorn_app`/`pg_trgm`/`GRANT ON SCHEMA public`)은 **모델링 이후**로 미룸.

**중요:** 인프라는 "붙는다"까지만 검증된 상태다. 모델링 단계에서는 인프라를 더 건드리지 않는다.

- success(초록) 토큰 추가 여부
- S2 '커뮤니티 근거' 소스 (권고: 내부 판매/스왑 통계 — 외부 커뮤니티 크롤링 아님)

## 8. 다음 작업 (우선순위)

1. **S1 최종 판정** — 브라우저에서 직접 클릭·타이핑. 대화 자연스러움/후보 풀 축소 체감/기어 전환 혼란 여부. 어색하면 그 자리에서 갈아엎기.
2. **S0 재정비** — S1 품질 기준(단색 아이콘·토큰 정합)으로 랜딩 목업 다시. 기존 S0는 풀컬러 아이콘·How박스 배경 누락 결함 있음.
3. **S2 목업** — 결과+근거+비교 한 장 제안서. 근거는 엔진 산출값 바인딩 전제(더미).
4. **S3·S4 목업.**
5. (모델링 완료·사용자 선언 후) 뒷단 착수: DB 유저 생성 → 카운트 API → 앱 라우팅 → 배포.

**작업 원칙:** 가장 불확실한 것부터, 이음새를 하나씩 검증. 한 번에 하나. 미정은 임의로 메우지 말고 질문.
